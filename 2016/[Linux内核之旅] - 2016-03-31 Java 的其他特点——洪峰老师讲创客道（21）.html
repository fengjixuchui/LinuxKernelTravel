在前面的一讲中我向大家介绍了Java FX8用于图形用户界面设计的一些情况，图形用户界面，对于应用程序的开发是极为重要的一个环节。

对于已经掌握了Java以前版本的同学，现在应该花大量的时间来研究Java FX8的一些新内容，逐渐把Swing忘记，因为Swing代码已经不再被R公司维护了，以后的图形界面设计语言只用Java FX8进行。对于还没有学习Java的同学的，我建议你一开始就学习Java FX8的图形用户界面的设计。

除了图形用户界面开发之外，一个应用程序肯定要完成某种或者某些特定的功能。所以，一旦图形用户界面的问题解决以后，就开始涉及到关于这些特定功能的实现问题，这是应用程序设计里面一个重要的内容。通常来讲在图形用户界面下，应用程序会向用户提供许多功能供用户使用，比如，在操作数据库时，在窗口最左边的栏目上可能有很多要操作的数据库；中间可能有一些对数据库的具体的选项（比如打开一个数据库、备份一个数据库）；窗口的最右边有可能是对数据库的内容进行显示或播放（比如视频可以播放出来，声音可以通过喇叭或扬声器播放出来），也可以在最右边的窗口下显示一些当时写的句式等；像这样的操作都说明在同一个用户下有许多不同的任务在并发执行。

在Java编程中，这样的一些任务可以通过Thread（多线程）来实现。从Java1.0的API开始就提供了对多线程的支持，而且在Java8里面对多线程的支持和Java1.0相比没有太大的变化。虽然在Java中初始化一个线程对象并不复杂，可以通过对Thread类的继承，或者通过支持一个Runnable接口来实现一个线程，然而，多线程的调度难度是非常大的，这是Java教学中的一个难点，也是大家学习中的一个难点，具体情况可以参考《黑客道》课程中的一些具体细节。当Java中的两个线程之间没有共享状态或者共享数据时，Java Thread线程是非常简单的，当两个或者是多个线程共享一些东西的时候，一定会有一个共享数据产生互斥，当互斥过度时会产生死锁。因此在Java多线程编程和操作系统内核设计中遇到的问题都可以应用到Java编程中，比如在1965年荷兰计算机科学家Dijkstra研究的哲学家就餐问题、读写问题和理发师睡眠问题等都可以在Java的多线程开发中派上用场。

在紫皮书的第三章讲对象、状态、模块的时候对多线程和面向对象编程之间的矛盾做了详细的讲解，大家可以回顾一下，在这一章的后面讲用Ada语言设计实时系统的时候还会回到这个课题上，在本书第六章讲操作系统以及在本书第七章讲进程的时候，会不同程度的涉及到多任务多进程和多线程问题。

前面在介绍Java语言技术特点的时候，已经讲过Java语言不支持指针运算，不支持指针意味着不能处理内存的分配与管理问题，所以对于驱动程序的设计Java是无能为力的。java语言本身是使用C语言开发的，即Java的虚拟机使用C语言开发的，所以Java语言提供了与C语言的接口（Java Native Interface ，简称JNI），这个规范是Java语言的一部分，有了JNI，可以方便的使用C语言去开发设备驱动程序。一旦用C语言写的设备驱动程序做好以后，就可以将C语言开发的模块打包到Java系统中，从而让Java也能操作设备驱动程序。相比用C或C++开发设备驱动程序或开发基于设备驱动的应用程序，我认为使用Java JNI更好一些，因为图形用户界面的设计用Java语言做是非常合适和高效的，只是关于设备驱动这部分用C写，然后驱动用Java图形用户界面打包，那么这种应用程序针对某个特定硬件的设备驱动程序的应用程序会加快开发速度。一般来讲用C语言开发设备驱动程序是可以的，当设备驱动程序非常复杂的时候也可以用C++开发，C++开发代码同样也可以通过JNI的规范打包到Java系统中。也可以把其他一些语言跟Java一起混合编程，比如用Java设计图形用户界面，而关于方法计算的部分可以用另一种语言做，比如句法分析和符号计算的部分可以用非常强大的LISP语言，可以让Java的图形用户界面接受用户的数据输入和一些事件，而真正进行符号计算的部分交给Rack程序去做，同理，因为Python语言擅长文本设计，所以可以将Java界面接受到的数据传递个Python去做。适合用什么语言开发的任务就用什么语言进行开发相应的算法部分。

那么如何让一个Java和另外一个程序进行通信呢？这个时候就需要用到Java的Network的API，Java联网的API和Java I/O的部分是密切相关的，但是Java的I/O设计比较复杂，如果没有经过系统的训练很难掌握Java的I/O部分。Java的I/O的API设计复杂的主要原因是Java要追求跨平台，为了跨平台，Java必须在I/O的处理上做一些取舍以及强化、简化的一些设计，这也是Java的I/O比较难学的一个原因。在Java I/O技术上发展起来的Java TCP/IP的联网功能设计的非常好，前面已经讲过，在1995年推出的Java语言提供的Java API1.0中TCP/IP的协议就已经出现了，也就是说，Java语言从诞生之日起就已经支持互联网的编程。在TCP/IP协议栈中，IP存在网络层，TCP和UDP都位于IP层之上，TCP协议是面向连接的，当建立一个通信过程时，必须先进行握手，建立连接之后双方再进行通信，通信完成后，双方说再见，这是一个连续的一个过程。UDP协议是基于数据报（Data Gram）的，一个Data Gram和下一个Data Gram之间没有相互的关联，UDP只保证发出去，对用户是否收到数据报以及收到的顺序是否是发送方发出的顺序，UDP一概不管。两个协议各有特点，也都有自己的应用。总体来看，TCP协议适用于传输可靠的数据，UDP协议用于传输非常短小的报文，这是两者技术上的差异。

无论是TCP的编程还是UDP的编程，都需要有客户机/服务器的概念。服务器的程序要监听IP地址和端口，IP地址和端口被称为Socket，在Socket要指定一个数据通信的协议，这是服务器端程序应该完成的功能。客户机程序也要在它的本机上找到一个IP地址和一个端口号构成一个Socket，客户机的Socket和服务器的Socket这一对Socket按双方约定的协议建立数据通信。设计TCP服务器的一个难点在于同一个服务器要处理多个来自网上的请求，在Java网络编程中，要理解阻塞服务器和非阻塞服务器两个概念，这与Java联网的API关系十分密切。阻塞服务器和非阻塞服务器的区别是：一旦客户机与服务器建立TCP连接后，服务器会监听它的端口并等待客户机发数据给它，如果客户机不向服务器法数据，服务器就会一直等待下去，这种服务器称作阻塞服务器。在Java1.4的API之前，所有Java的TCP网络通信都是按照阻塞式的方式进行设计的。这种阻塞式的服务器对低负载的网站工作没有问题，因为低负载的网站上请求的数据不多，但是对于高负载的网站，阻塞式的服务器编程模型就无法应对了。从Java1.4开始，Java新的Network的API（NIO的API）会引入进来，这个API的引入是为了让Java去支持高负载、高并发的网络应用程序的设计而提供的。在非阻塞式服务器的设计中，需要理解Buffer和Channel这两个概念，Buffer是缓冲区，Channel是把同一个连接按照某种数据格式细分出来，这样对信道的占用变小了，相当于把原来的一条单车道马路分成双车道、三车道或更多的车道，每一个车道上可以跑一辆车，这样的话可以同时在一个连接上跑多辆车。这是一个基本的设计思想，而且对于每一个Channel上抛出的数据首先要放到Buffer里面，这两个概念支撑起了Java NIO的API的基础，有了Buffer和Channel这两个概念之后，Java NIO的API还提供了Selects（选择器）。如何从很多的Buffer里面进行选择数据投放给后台的进程，这个问题在Java NIO中解决的非常好，也是我们设计高性能网络服务器的一个首选的API，从Java7开始有一个Java NIO.2的API被引入进来，NIO.2的API对文件系统的操作简化了。

到此为止，对于利用Java语言设计应用程序的细节又向前深入了一步，已经介绍了用Java语言设计图形用户界面、用Java的线程去支持并发以及Java的联网功能，还有让Java和许多编程语言进行配合，发挥各种语言的功能。有了这些编程的技巧以后，对Java语言设计应用程序在生产力上会有大幅度的提高。
