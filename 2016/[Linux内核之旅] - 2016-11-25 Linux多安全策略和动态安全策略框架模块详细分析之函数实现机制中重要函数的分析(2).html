2. security_compute_av()

security_compute_av()函数用于根据客体类别及SID对(ssid，tsid)来计算访问向量决策。函数头如下所示：

void security_compute_av(u32 ssid,u32 tsid,orig_tclass,struct av_decision *avd)


该函数包含4个参数：ssid表示源安全标识符；tsid表示目标安全标识符；tclass表示目标安全类别；avd表示访问向量决策。对于该函数，其函数调用流程图如图2-11所示：


图2-11 security_compute_av()函数调用流程图


下面结合源码对该函数的主要执行步骤进行说明：

①　调用avd_init()函数来初始化相应的访问向量决策。

②　判断ss_initialized变量是否设置，如果没有，表示安全服务器还没有进行初始化，此时直接返回。

③　调用sidtab_search()函数分别查找ssid和tsid对应的安全上下文结构scontext和tcontext结构，如果没有找到则直接返回。

④　调用context_struct_compute_av()函数来根据客体类别和安全上下文对计算访问向量。对于该函数下面会进行详细分析。

⑤　调用map_decision()函数根据客体类别来获取相应的访问向量决策。

⑥　结束并返回。

 

如上所述，security_compute_av()函数主要通过context_struct_compute_av()函数来计算访问向量决策，下面主要对该函数进行分析。context_struct_compute_av()函数用于根据客体类别和安全上下文对来计算访问向量决策，函数头如下所示：

static void context_struct_compute_av(struct context *scontext,

   struct context *tcontext,u16 tclass,

   struct av_decision *avd)


该函数包含4个参数：scontext表示源安全上下文；tcontext表示目标安全上下文；tclass表示客体类别；avd表示访问向量决策。



图2-12 context_struct_compute_av()函数调用流程图


如图2-12所示，下面结合源码对该函数的主要执行步骤进行说明：

①　判断参数tclass的有效性，如果参数无效，则直接返回。

②　调用flex_array_get()函数分别根据源和目标上下文中的类型获取相应的属性。

③　调用ebitmap_for_each_positive_bit()宏，该宏实际上是一个for循环。这里两次调用该宏分别对源和目标属性进行组对，然后以其作为键值来查找策略中的访问向量规则，如果存在相应的规则，则根据规则的类型设置访问向量决策，接着调用cond_compute_av()函数根据相应的键值来检查条件访问向量表来确定是否授予了额外的权限，并对访问向量决策进行设置。

④　采用while循环来遍历客体类别的权限的约束，并根据策略库中的布尔值调用constraint_expr_eval()函数来计算相应的条件策略的条件表达式，然后根据相应的规则删除访问向量决策中的权限。

⑤　判断策略库是否支持角色转换规则。如果支持，则通过for循环查找相应的角色转换规则，如果找到，则进行后续操作；反之从访问向量决策中删除策略指定的转换权限。

⑥　调用type_attribute_bounds_av()函数删除违反边界约束的权限。

⑦　结束并返回。

 

对于type_attribute_bounds_av()函数，其用于删除违反边界约束的权限。函数头如下所示：

static void type_attribute_bounds_av(struct context *scontext,struct context *tcontext,        u16 tclass,struct av_decision *avd)  


该函数包含4个参数，其含义和context_struct_compute_av()函数的参数含义相同，这里不再赘述。对于该函数，其函数调用流程图如图2-13所示：


图2-13 type_attribute_bounds_av()函数调用流程图


下面结合源码对该函数的主要执行步骤进行说明：

①　调用flex_array_get_ptr()函数获取源安全上下文和目标安全上下文中的类型属性。

②　判断源安全上下文的类型属性的边界是否设置。如果设置，则将源安全上下文拷贝到新的上下文结构lo_scontext中，并用源安全上下文的边界替换新的安全上下文的类型字段，然后通过context_struct_compute_av()函数根据新的安全上下文，目标安全上下文，目标类别计算一个访问向量决策lo_avd。如果lo_avd中的允许的权限和avd中的允许的权限的超集，表示没有要删除的权限，此时函数直接返回；反之则获取avd中不允许的权限，并存放到masked中。

③　判断目标安全上下文的类型属性的边界是否设置。其处理过程和源安全上下文的处理过程相同，因此这里不再赘述。

④　如果源和目标安全上下文均设置了类型边界，则根据相应类型的边界来修改对应的安全上下文中的类型属性来获得新的源和目标安全上下文，接着根据这两个安全上下文调用

context_struct_compute_av()函数从而计算相应的访问向量决策lo_avd。如果lo_avd中的允许的权限和avd中的允许的权限的超集，表示没有要删除的权限，此时函数直接返回；反之则获取avd中不允许的权限，并存放到masked中。

⑤　如果masked不为空，表示某些权限不允许，此时调用security_dump_masked_av()函数审计相应的权限。对于security_dump_masked_av()函数，主要调用了一些审计相关的函数，如audit_log_start()、audit_log_format()等，对于这些函数，参见“Linux安全审计机制模块代码分析报告”一文。

⑥　结束并返回。
