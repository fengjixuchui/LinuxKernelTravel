什么是CFS带宽控制

在了解CFS带宽控制之前，首先要知道的是CPU是一种可被管理的资源，也就说它是可以被精细化分配的资源。CFS带宽控制在很多场景里都很有用，一方面在它出现之前，经常会发生某个任务使用过量CPU导致其它任务出现不可接受的延迟或者抖动的现象。另一方面比如在一个多租户的场景里，系统（云平台）要按照用户对各种资源的使用量进行收费，这时就需要对CPU的上限进行控制，以免用户超额使用。


为了实现这个CPU带宽控制的目标，Paul Turner和Bharata Rao先后提出了对CFS带宽控制的内核补丁，在CFS bandwidth control中对他们的工作有相关的描述。按照CFS Bandwidth Control的描述，CFS带宽控制是用来限制一个任务组能够消耗的CPU的上限，具体来说就是通过设置period和quota，指定在period时间里该任务组最多能够消耗quota的CPU时间。

技术原理

在谈到具体实现之前，还需要明确几个任务调度相关的概念。从2.6内核以后，为了缓解多用户下进程调度不公平的情况（举个例子：A用户有个实时进程，B用户有个非交互进程，因为A的优先级比较高，那么A占用了大量的CPU时间，那么对B就不够公平）被调度的对象就从单一进程变成了任务组，这样一来可以每个用户对应一个任务组，从而分配得到一定的CPU时间，在这个组里不同的任务再去获取相应的CPU时间。这样的任务组叫做task_group,

定义在kernel/sched/sched.h:363中:

struct task_group {#ifdef CONFIG_FAIR_GROUP_SCHED // 由该参数控制该特性
/* schedulable entities of this group on each CPU */struct sched_entity **se; // 执行实体/* runqueue "owned" by this group on each CPU */struct cfs_rq       **cfs_rq;#endifstruct cfs_bandwidth    cfs_bandwidth; // CPU带宽}

其中包含了一个类型为sched_entity的成员。sched_entity也即调度实体，它可以是进程，也可以是进程组，甚至用户。这些调度实体按照vruntime时间序，以红黑树的形式组织，并由cfs_rq管理，同样是task_group的字段，它的定义在：kernel/sched/sched.h:488：

struct cfs_rq {#ifdef CONFIG_FAIR_GROUP_SCHEDstruct rq       *rq;    /* CPU runqueue to which this cfs_rq is attached */};

它包含了一个rq结构的成员，这是该cfs_rq附着到每个CPU run queue的指针。


在回到task_group的定义里，还可以看到cfs_bandwidth的结构体成员，它的定义在：`kernel/sched/sched.h:337`：

struct cfs_bandwidth {#ifdef CONFIG_CFS_BANDWIDTHktime_t     period;   // periodu64     quota;        // quota#endif};

可以看到period和quota正是cfs_bandwidth的成员。那么综上所述，CPU带宽控制的对象是task_group，配置好的period和quota都是作用在task_group上，然后再分配给task_group下的cfs_rq。可以以下图作为示例:


进一步说，可以认为给task_group分配的quota是一个global quota pool，然后假定每个cfs_rq又拥有一个local quota pool，那么cfs_rq要从global quota pool中申请一定数量的CPU时间，这个时间叫做slice,可以通过/proc/sys/kernel/sched_cfs_bandwidth_slice_us，目前默认值为5ms。见下图：


当某个cfs_rq申请到slice以后就会开始执行，当前slice消耗完后就会继续申请。当在一个period的时间global quota pool已经分配干净，此时该cfs_rq就会进入一个叫做throttled的状态，在这个状态中任务因为无法获取CPU时间所以无法被执行。这个throttle的统计状态可以通过cpu.stat cgroup进行查看。当该period结束后，global quota pool会被刷新并可以分配slice，如此进入下一个周期。


那么回到前面所说的cfs_bandwidth定义里：

struct cfs_bandwidth {#ifdef CONFIG_CFS_BANDWIDTHktime_t     period;   // periodu64     quota;        // quotau64     runtime;      // 剩余时间，随任务执行减少
struct hrtimer    period_timer; // 高精度周期定时器，与period时间相等struct hrtimer    slack_timer;  // 实现延迟归还struct list_head  throttled_cfs_rq;/* Statistics: */                // cpu.stat的返回值int     nr_periods;          // 经历的时段数，单位为次数int     nr_throttled;        // 发生throttle的次数u64     throttled_time;      // 被throttle的时间总和#endif};

可以了解到period的周期性由一个高精度定时器period_timer实现，被throttle的cfs_rq会挂到throttle_cfs_rq链表中。runtime的值最初跟quota一样，但随着任务执行会逐步减少。cpu.stats的值来自厦门的nr_periods、nr_throttled和throttled_time。

实验测试

在Google Cloud上申请一台单核的服务器进行测试，代码采用

[cfs.go(https://gist.github.com/bobrik/2030ff040fad360327a5fab7a09c4ff1#file-cfs-go)。

这段代码的主要功能是执行一个可以造成CPU密集的sha512加密算法，并执行5ms，然后进入sleep状态，待sleep结束以后继续进行sha512加密，如此循环。通过参数可以配置sleep的时间和循环次数iteration。


为了方便测试，采用go 1.12.5的Docker镜像进行测试，首先看一下不开启带宽控制的时候，执行下面的命令：

`docker run --rm -it -v $(pwd):$(pwd) -w $(pwd) golang:1.12.5 go run cfs.go -iterations 20 -sleep 10ms`

得到结果：


可以从"burn took"看出来执行加密只需要5ms的时间，跟代码里写的是一样的。


然后设置quota为25ms，period为100ms，程序slice是系统默认的5ms。比如执行下面的命令：

docker run --rm -it --cpu-quota 25000 --cpu-period 100000 -v $(pwd):$(pwd) -w $(pwd) golang:1.12.5 go run cfs.go -iterations 20 -sleep 10ms

得到结果：


这个时候就会看到"burn took"有的时候会花费25ms，这个时间是怎么来的？当任务开始执行的时候，先执行5ms，然后睡眠10ms，然后又执行5ms                                                                                   



也就是说当25ms的quota耗尽的时候，总共经历了75ms（忽略第一个period）。此时由于quota已经耗尽，所以无法执行。然后从这一刻起直到period要刷新的第200ms，该cfs_rq就处于throttle的状态。这部分throttle的时间刚好就是25ms。
