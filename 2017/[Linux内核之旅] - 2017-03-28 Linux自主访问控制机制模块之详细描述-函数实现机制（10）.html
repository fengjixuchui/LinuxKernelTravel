ext4_xattr_set_acl()

ext4_xattr_set_acl()函数用于从扩展属性中获取指定文件的内存表示形式的ACL，并用其设置目标文件的指定类型的ACL属性，函数头如下所示：


static int ext4_xattr_set_acl(struct dentry *dentry, const char *name, const void *value,size_t size, int flags, int type)


该函数包含6个参数：dentry表示目标文件对应的目录项，name表示扩展属性的名字；buffer表示存放扩展属性的缓冲区；size表示缓冲区的长度；flags表示相应的操作标志，在该函数中未使用该参数；type表示ACL的类型。如图2-23所示，该函数主要是通过posix_acl_from_xattr()函数来实现的，而posix_acl_from_xattr()函数在前面已经分析过了，所以这里不对其进行详细说明。

ACL的访问控制过程

ACL作为一种访问控制机制，那它是如何实现权限的限制的呢？换句话说，当用户访问某个文件时，如何使用ACL机制来判断该用户是否具有访问该文件及该文件所在的目录的权限？这看似很复杂，因为首先要检查用户是否可以访问该文件所在的目录，如果可以，接着还要判断该用户能否访问相应的文件。由于目录本质上也是一个文件，因此对目录权限的检查就转化成了对文件的权限的检查，所以上述问题就归结为如何判断用户可以访问指定的文件。


inode_permission()

对于指定的文件，其存放路径是已知的，那么如何由指定的路径获得相应的文件呢？在linux3.5.4中，该过程是通过path_openat()函数实现的，path_openat()函数调用了两个重要的函数path_init()和link_path_walk()，后者用于名字的解析。而link_path_walk()函数调用了may_lookup()函数，may_lookup()函数调用了inode_permission()函数，经过分析得知inode_permission()函数就用于文件访问权限的检查。实际上may_lookup()函数就是通过调用inode_permission()函数来检查相应文件是否具有执行权限，因为只有进程拥有对目录的执行权限，进程才能进入该目录，从而来执行其他的操作。对于inode_permission()函数，主要用于检查当前进程是否具有指定inode的指定权限，其定义在fs/namei.c中，函数头如下所示：


int inode_permission(struct inode *inode, int mask)


该函数有两个参数：inode表示要进行权限检查的目标文件；mask表示待检查的权限。

 

图2-24 inode_permission()函数调用流程图


如图2-24所示，下面结合源码对该函数的执行步骤进行说明：

①　判断mask中是否指定了MAY_WRITE，如果指定，则表示检查写权限。对于只读文件、普通文件、目录及链接文件之外的其他文件，不允许进行写操作；另外如果一个文件指定了S_IMMUTABLE标志，表示该文件为不可改变的文件，那么该文件也不允许进行写操作。

②　调用do_inode_permission()函数，该函数主要通过调用generic_permission()函数来进行权限的检查，这里主要基于ACL进行权限的检查。

③　调用dev_cgroup_inode_permission()函数，该函数根据cgroups机制的设定来判断当前进程是否可以访问指定的设备，即目标文件对应的设备。

④　调用security_inode_permission()函数，该函数根据lsm模块实现的安全策略来进行权限的检查。
