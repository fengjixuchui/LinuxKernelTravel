本学期在陈老师的带领下，第一次踏进内核的殿堂，第一堂课就是阅读list.h。


首先交代下我阅读的内核版本是linux-4.18.7。list.h是内核中的一个头文件，主要有两大部分：以list_ 命名的双链表相关的操作和以hlist_命名的哈希表的相关操作。我这里主要记录的是hlist部分一些我觉得有趣的地方与大家分享。

哈希表：

哈希表简单的说就是将关键字与其存储位置建立一种函数关系。实现哈希表有两个关键的步骤：

一是构造哈希函数。

二是解决冲突问题。

哈希函数的构造因人而异，而解决冲突则是每个哈希表需要面临的问题。

内核为我们提供了用于实现链地址法的各种数据结构，函数和宏，有了它们我们可以很方便的造出哈希表。这里体现了一种抽象的思想。

  • 我们先来看看内核中对于哈希表结构体的定义：


    struct hlist_head {
        struct hlist_node *first;
    };

    struct hlist_node {
        struct hlist_node *next, **pprev;
    };

    这段代码在types.h中。内核中对于哈希表，定义了两种节点，一个是表头的_head节点，它只有一个指向第一个链接点的first指针。另一个则是用于组成链表的_node节点。

    有趣的是这个_node节点，它不像一般的单链表只有一个指向下一节点的next指针，也不像双链表那样有一个指向前驱节点的pre指针。 它有一个独特的二级指针pprev，它是指向哪里的呢?

    我们接着往下看。

  • 为了了解hlist_node中二级指针的作用，我们去list.h中找找答案


    • #define HLIST_HEAD_INIT { .first = NULL }                  
      #define HLIST_HEAD(name) struct  name = {  .first = NULL }  //声明并初始化
      #define INIT_HLIST_HEAD(ptr) ((ptr)->first = NULL)  

      几个宏都是对_head进行初始化的宏，在双链表list_也有类似的写法，没有我们想要的二级指针。


    • static inline void INIT_HLIST_NODE(struct hlist_node *h)
      {
          h->next = NULL;
          h->pprev = NULL;
      }

      初始化hlist_node的函数，将_node的连个指针都指向NULL。还是看不出什么。


    • static inline void __hlist_del(struct hlist_node *n)
      {
          struct hlist_node *next = n->next;
          struct hlist_node **pprev = n->pprev;

          WRITE_ONCE(*pprev, next);
          if (next)
              next->pprev = pprev;
      }

      删除某一hlist_node节点的函数。看看我们找到了什么：

      WRITE_ONCE(*pprev, next);

      这不正是对pprev进行操作的语句吗。

      我们知道，链地址法就是将产生冲突的若干数据用链表串起来存储，那么不论是单链表，双链表还是其他的什么，它总归是个链表，对其中节点的删除也无外乎对其前驱后继节点指针的修改。

      而二级指针就是一个指向指针的指针，*pperv 一次指向运算表示它指向的那个指针，**pperv连次指向运算则表示它指向的指针所指向的地址。

      再来看代码，首先建立的两个指针next和pprev保存了当前节点指针的指向。

      而write_once(a,b) 即是a=b；

      这样看来我们的pprev指向的那个指针，就是用来指向当前节点的后继；

      最后一步是当后继节点不为空时，将其pprev指向当前节点的pprev；

      结合我们对一般链表的删除操作，当我们删除一个节点时，用来指向其后继的应当是其前驱的next指针。所以pprev正是一个指向前驱节点next指针的指针。

  • 搞明白了hlist_node中的指针，我们终于可以对内核中的哈希表有一个形象的概念了。

    现在，有个问题需要我们仔细想想，为什么hlist_node要这样设计？

    • 只用一个单链表不好吗？用单链表把冲突的节点连起来是没什么问题，但是对其进行操作就很烦了，插入还好说，若是删除某个特定节点，还需要从头遍历找到它的前驱才行。

    • 那用双链表不就行了吗？确实，用双链表可以很方便的进行插入删除等操作，而且list.h中也有现成的函数和宏可以使用。但是，我们对比下hlist和list就能发现它两最大的区别就在其节点的定义上。list的结构体只有一个节点list_node，而hlist则是两种hlist_head和hlist_node。假设我们的hlist_node也是一个双链表结构，有一个指向前驱的pre指针，那么这个指针在指向hlist_head还时需要进行一次类型转换，所以在每次操作前都需要判断其指向节点的类型，这很是麻烦。

    • 那索性把hlist_head也改成hlist_node不就完了么。

      这样一改后的确不再有上述的麻烦，但是又有新的问题，我们的“头结点”需要这个pre指针吗？哈希表中的“头节点”不存放数据，只用来指向相应链表。给它多增加一个无用的指针，相当于将其空间占用提高了一倍，哈希表中头结点数量与数据总数是在一个数量级上，这将造成巨大的浪费。

    • 综上，linux内核哈希表无疑是一个非常优秀的设计。

  • list.h中让我很费解的函数

    为哈希表增加一个虚假节点和判断一个节点是否是虚假的函数。


    static inline void hlist_add_fake(struct hlist_node *n)
    {
        n->pprev = &n->next;
    }

    static inline bool hlist_fake(struct hlist_node *h)
    {
        return h->pprev == &h->next;
    }

    我只能从语句上试着理解其作用，hlish_add_fake函数将一个节点的pprev指针指向自己的next指针，但这个节点仍可能在hlish节点链表上，从而成为一个”虚假节点“。

    而这样的”虚假节点“ 仍会被 hlist_unhashed 判断为已 hash。刚经过初始化的节点，其pprev指向NULL，即未hash。


    static inline int hlist_unhashed(const struct hlist_node *h)
    {                            
        return !h->pprev;         
    }

    而对它进行 hlist_del操作则不会将其从链表上删除，如下图。其他的用法还不是很清楚。

  • 遍历宏


    #define hlist_for_each(pos, head) 
        for (pos = (head)->first; pos ; pos = pos->next)

    #define hlist_for_each_safe(pos, n, head) 
        for (pos = (head)->first; pos && ({ n = pos->next; 1; }); 
             pos = n)

    两者的区别与双链表中类似，hlist_for_each_safe 可以在遍历的过程中对节点进行添加/删除操作，而前者则不能。

    值得一提的是hlist_for_each_safe中的 ({ n = pos->next; 1; }) 它是一条复合语句。C语言中复合语句：({exp1; exp2; exp3})  它的值等于exp3的值。如果exp3不能求值，得到的结果就是void。复合语句在内核代码中的宏定义中很常见。

    通过一个复合语句将赋值语句n=pos->next写在for循环的判断条件中，使代码更简洁。

关于hlist大概就是这些，下面我们看看另一个有趣的宏。

ACCESS_ONCE()宏

  • 在阅读list.h的过程中，经常碰到两个宏WRITE_ONCE(a,b)和READ_ONCE(x) 。

    <linux / compiler.h>中找到宏的定义


    #define WRITE_ONCE(var, val) 
        (*((volatile typeof(val) *)(&(var))) = (val))
        
    #define READ_ONCE(var) (*((volatile typeof(var) *)(&(var))))

    直接看宏的定义就很复杂，就READ_ONCE(x)而言，它执行了取地址，强制转换成某种指针类型，再指向这个指针。从代码上来看，WRITE_ONCE(a,b)就等价于a=b，READ_ONCE(x)就是读取x的值。为什么要把简答的语句弄的这么复杂，这背后的原因非常令我好奇。在网上查阅了一些资料后，对这个问题有了一个大致的了解。

  • 首先WRITE_ONCE(a,b)和READ_ONCE(x)都属于ACCESS_ONCE()宏。在LWN上有一篇专门对该宏的介绍（https://lwn.net/Articles/508991/）。

    该宏的目的确保作为参数传递的的值仅由生成的代码访问一次。如果不是，C编译器则会假定它正在编译的程序的地址空间中只有一个执行线程，通俗的说就是C编译器认为该程序只会由一个线程来执行。这会导致一些并发性上的问题，而C语言本身并没有并发的概念。所以需要在C语言之上构建处理并发访问的机制，ACCESS_ONCE()宏就是这样一种机制。

    而为了实现这种机制，则需要用到volatile 变量。关于volatile变量，维基百科中解释的很清楚：

    volatile关键字声明的变量或对象通常具有与优化、多线程相关的特殊属性。通常，volatile关键字用来阻止（伪）编译器认为的无法“被代码本身”改变的代码（变量/对象）进行优化。如在C语言中，volatile关键字可以用来提醒编译器它后面所定义的变量随时有可能改变，因此编译后的程序每次需要存储或读取这个变量的时候，都会直接从变量地址中读取数据。如果没有volatile关键字，则编译器可能优化读取和存储，可能暂时使用寄存器中的值，如果这个变量由别的程序更新了的话，将出现不一致的现象。

    一个简单的例子可以帮助我们理解volatile的作用。

    在这里例子中，代码将foo的值设置为0。然后开始不断地轮询它的值直到它变成255：


    static int foo;
     
    void bar(void) {
        foo = 0;
     
        while (foo != 255)
             ;
    }

    一个执行优化的编译器会提示没有代码能修改foo的值，并假设它永远都只会是0.因此编译器将用类似下列的无限循环替换函数体


    void bar_optimized(void) {
        foo = 0;
     
        while (true)
             ;
    }

    但是，foo可能指向一个随时都能被计算机系统其他部分修改的地址，例如一个连接到中央处理器的设备的硬件寄存器，上面的代码永远检测不到这样的修改。如果不使用volatile关键字，编译器将假设当前程序是系统中唯一能改变这个值部分。 为了阻止编译器像上面那样优化代码，需要使用volatile关键字：


    static volatile int foo;
     
    void bar (void) {
        foo = 0;
     
        while (foo != 255)
            ;
    }

    这样修改以后循环条件就不会被优化掉，当值改变的时候系统将会检测到。

  • 然后，结合的ACCESS_ONCE()宏的定义：先取x的地址，将其强制转换为 x 类型的volatile 指针，再指向该指针。


    #define ACCESS_ONCE（x）（*（volatile typeof（x）*）＆（x））

    下面这种写法可能看的更清楚：


    volatile typeof(x) *p = &(x);
    *p;

    ACCESS_ONCE()宏的工作原理就是将相关变量的类型临时转换为volatile类型，直接对其内存地址进行访问。

  • 最后，还有一个问题：在同一个函数内，同样的赋值操作，为什么有的地方可以直接赋值，而有的地方却需要用到ACCESS_ONCE()宏？

    LWN中是这样写的：

    大多数并发访问数据是（或应该是）由锁保护。如果代码只访问具有相关锁的共享变量，并且该变量只能在释放锁（并由另一个线程持有）时更改，则编译器不会产生问题。只有在没有锁的情况下访问共享数据的地方才会需要像ACCESS_ONCE（）这样的宏。

    那么新的问题又来了，如何找到这些没有上锁的共享数据？
